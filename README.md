# BO26EB-06 Gazebo Hardware in the loop simulator

Simulation workspace for a Palfinger crane mounted on a platform, with operator training scenarios, HMI range monitor, crane cameras, payload snap tooling, and Gazebo-based marine motion.

## What This Repo Contains

- Crane simulation with slew, trolley, winch, sway, drum, and wire visualization joints
- Havyard ship model with wave / motion tooling
- Crane camera suite with boom-tip, hook, and cabin views
- Operator training terminal with scenario shortcuts and priority override lane
- Snap target discovery and attach / detach workflow for training payloads

## Current Status

The current setup supports:

- Manual crane teleoperation
- Terminal-based operator training scenarios
- Priority command override for training mode
- Split crane bridge architecture for slew and reach control
- Camera streams, the interactive camera viewer, and HMI range monitoring
- Blue container snap targets and snap state feedback
- Ship DP hold and synthetic wave motion for the Havyard vessel

Current limitation:

- The snap workflow does not yet create a true physical welded attachment in Gazebo. It is still training-oriented hook-follow behavior rather than a full rigid physical attach.
- If a snapped container is released high in the air and then falls under physics, the snap target may still represent the previous snap/release pose. For repeatable pickup training, release close to the intended surface.

## Recent Improvements

Recent updates focused on making the simulator more stable and easier to operate during lifting and training:

- Sim-facing ROS 2 nodes now use `use_sim_time`
- Control and camera topics now use explicit ROS 2 QoS profiles instead of the generic queue depth `10`
- Snap is enabled by default in the main launch file
- Snap candidate selection is now restricted to attachable container targets
- Snapped hook clearance was reduced so the payload sits closer to the hook during lifting
- Temporary visual snap-area marker on the container was hidden without changing the logical snap area
- Hook camera mount and Hugin platform collision alignment were tuned for better operator view and contact behavior

Main benefits:

- More consistent timeout and timer behavior when Gazebo pauses, slows, or starts up
- Less stale command and camera data under load
- More predictable snap behavior when operating near the container
- Better visual alignment between platform deck and collision surface
- Cleaner hook camera framing while lifting payloads


## Run

Launch the main simulation with:

```bash
cd ros2_ws
source /opt/ros/jazzy/setup.bash
source install/setup.bash
ros2 launch palfinger_sim sim.launch.py
```

Current launch defaults:

- `command_source:=joystick`
- `enable_snap:=true`
- `enable_camera_viewer:=true`
- `enable_hmi_view:=true`
- `enable_waves:=true`
- `enable_dp_hold:=true`
- `ocean_wave_profile:=calm`
- `sea_state_profile:=calm`
- `wave_amplitude:=-1.0`
- `wave_period:=-1.0`
- `wave_direction:=90.0`

Wave-related launch arguments mean different things:

- `sea_state_profile`
  Controls the synthetic ship motion generated by `ship_dp_hold.py`.
  This affects how the vessel actually moves in roll, pitch, and heave.
- `ocean_wave_profile`
  Controls which visual ocean world is loaded.
  This affects how rough the sea looks in Gazebo, but not by itself how the ship motion node behaves.
- `wave_direction`
  Controls the synthetic wave direction used by `ship_dp_hold.py`, which changes how motion is distributed between roll and pitch.

Common combinations:

- `ocean_wave_profile:=calm sea_state_profile:=calm`
  Calm visuals and calm ship motion.
- `ocean_wave_profile:=rough sea_state_profile:=rough`
  Rough visuals and rough ship motion.
- `ocean_wave_profile:=rough sea_state_profile:=calm`
  Rough-looking sea with gentle ship motion.

## Useful Launch Modes

The main launch supports different operator input modes:

- `command_source:=joystick`
- `command_source:=terminal_rate`
- `command_source:=terminal_target`
- `command_source:=terminal_operator_training`

Example:

```bash
ros2 launch palfinger_sim sim.launch.py command_source:=terminal_operator_training
```

Other useful launch examples:

```bash
ros2 launch palfinger_sim sim.launch.py enable_camera_viewer:=false enable_hmi_view:=false
ros2 launch palfinger_sim sim.launch.py ocean_wave_profile:=rough sea_state_profile:=rough
ros2 launch palfinger_sim sim.launch.py ocean_wave_profile:=rough sea_state_profile:=rough wave_direction:=45.0
ros2 launch palfinger_sim sim.launch.py command_source:=terminal_target
ros2 launch palfinger_sim sim.launch.py training_profile:=gusty_wind
```

## Operator Training

The operator training terminal provides quick scenario shortcuts and uses the priority lane.

Behavior depends on mode:

- `operator_training` mode publishes priority disturbance while normal joystick control can still drive the crane
- `rate` mode publishes priority rate commands
- `target` mode publishes priority target commands

Common scenarios:

- `1` gusty wind
- `2` sway training
- `3` recovery
- `4` pendulum kick
- `5` hoist snag release
- `0` off

The training node also listens on `/crane/training_command`, so scenarios can be triggered externally in addition to terminal input.

## Cameras

The crane includes:

- boom-tip camera
- hook camera
- cabin camera
- joystick-driven pan / tilt control bridged into Gazebo for all three cameras
- the default `camera_viewer` window when `enable_camera_viewer:=true`
- HMI range monitor and viewer when `enable_hmi_view:=true`

Recent camera-related tuning:

- Image transport now uses sensor-data QoS on the ROS 2 side so old frames are dropped instead of queued
- The hook camera mount point was moved slightly upward to reduce how close the payload appears during lifts

The standalone OpenCV-based `camera_viewer` node starts automatically by default with `enable_camera_viewer:=true`. The launch also starts `hmi_range_monitor` and `hmi_range_viewer` by default for operator distance and payload-state feedback.

## Worlds

Available world variants are in `ros2_ws/src/palfinger_sim/worlds/`.

The main launch currently selects between:

- `ocean_wave_profile:=calm`
- `ocean_wave_profile:=moderate`
- `ocean_wave_profile:=rough`

Additional world files such as evening and backup/test variants exist in the repository but are not selected by dedicated launch arguments by default. They can still be loaded explicitly with `world_sdf:=/path/to/world.sdf`.

## Snap

Snap support now starts by default in the main launch.

Current snap behavior:

- `snap_target_provider` publishes ship and container snap targets
- `snap_manager` only selects attachable container targets by default
- `snap_executor` follows the hook by repositioning the attached container model through Gazebo services
- the visible debug snap-area marker on the container model is disabled, but the logical snap hitbox remains active
- the main launch currently spawns `container_blue`; the extra AA/DPA/SS container models are present in the repository, but their spawn nodes are commented out in `sim.launch.py`

Current limitation:

- this is still a training-oriented follow attach, not a rigid welded physics joint
